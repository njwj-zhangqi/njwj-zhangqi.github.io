@RequestMapping在同一个Controller中，不能完全一样

Ajax参数的说明：

  • url : 请求的URL
  • data ： 发送给服务器的数据
  • type ： 请求类型，有GET，POST等
  • contentType : 请求内容的类型，有formdata和payload两种方式
  • dataType ：返回的数据类型
  • success ：成功返回的回调函数

Controller注解的说明：

  • @RequestMapping，在方法上，URL映射路径，与ajax的url相对应
  • @RequestBody，在参数中，接收针对payload方式传递的参数,常用于JSON格式数据,只接收JSon对象的字符串
  • @PathVariable，接收针对路径中的参数
  • @RequestParam，针对普通的参数传递，不想使用POJO去接收，或者转换参数的名称
  • @ResponseBody，不加此注解，如果返回String，则会被解析为jsp文件，加上后正常返回String

bean的作用域

  • 单例（Singleton），只创建一个实例
  • 原型（Prototype），每次注入或者通过Spring应用上下文获取的时候，都会创建一个新的bean实例

会话作用域的作用：每一个会话创建一个实例，但是当要注入到Service中的时候，Service是单例的，而且多个实例注入到单例Service中，可能会造成失效问题。所以需要使用proxyMode，在需要注入的时候，会生成一个代理对象注入到Service中，在调用时，代理对象会调用真实的会话实例方法。代理的是接口，则使用ScopedProxyMode.INTERFACES,如果是类，使用ScopedProxyMode.TARGET_CLASS。

SpringEL的特性：

  • 使用bean的id来引用bean
  • 调用方法和访问对象的属性

SpringEL表达式使用 #{…}，和占位符 ${…}不同。

  • T() 表达式 会将括号中的类型视为Java中对应的类型，如T(System)，则可以写出#{ T(System).currentTimeMillis()}，表示当前时间，还有#{T(java.lang.Math).PI}，引用常量或静态方法
  • 引用bean或者bean的属性，如: #{braveKnight.name}，调用bean的方法#{knight.call()},还可以根据返回值类型连续调用#{knight.call().toUpperCase()},避免为null，使用#{knight.call()?.toUpperCase()}，如果call()返回的是null，则不会调用右边的方法，值就是null
  • 引用系统属性#{systemProperties[‘disc.title’]}
  • 比较数字相等#{knight.year == 100},或者文本型的eq，#{knight.year eq 100}
  • 三元表达式${disc.title > 100 ? “Yes” : “no”}，简化版，常用的判空处理#{disc.title?: ‘Rattle and Hum’}
  • 对集合和数组的操作，#{list[1]}，查询运算符.?[]，会对集合进行过滤，得到集合的一个自己。#{songs.?[artist eq ‘AAAA’] }，.^[]查找第一个匹配项，.$[]查找最后一个匹配项，.![]投影运算符#{songs.![title]}，将songs里面的所有title重新投影

在Servlet3.0环境中，容器在类路径下查找实现javax.servlet.ServletCotainerInitializer接口的类，能发现，就用来配置Servlet容器。

Spring则提供了上述接口的实现类，叫SpringServletCotainerInitializer，这个类会反过来查找实现WebApplicationInitializer的类，并将配置的任务交给它们来完成。Spring3.2引入一个基础实现AbstractAnnotationConfigDispatchServletInitializer。，因此只要继承基础实现类，就能配置Servlet上下文。

重写三个方法：getServletMapping(),将一个或多个路径映射到DispatchServlet上，一般映射为”/“

当DispatchServlet启动时，会创建Spring应用上下文，并加载配置文件或配置类中所声明的bean，所以由getServletConfigClasses中来进行配置

在Spring Web应用中，通常会有另外一个应用上下文，由ContextLoaderListener创建的，通过getRootConfigClasses中来进行配置。

我们希望DispatchServlet加载包含Web组件的bean，如控制器、视图解析器以及处理器映射，而ContextLoaderListener要加载应用中的其他bean，通常是驱动应用后端的中间层和数据层组件。

通过@EnableWebMvc启用注解驱动的Spring MVC

  • 没有配置视图解析器。Spring会默认使用BeanNameViewResolver，会查询Id与视图名称匹配的bean，并且查找的Bean要实现View接口
  • 没有启用组件扫描，Spring只能找到显示声明在配置类中的控制器
  • DispatchServlet会映射为应用的默认Servlet，对所有请求进行处理，包括静态资源。

处理Multipart形式的数据,先配置一个multipart解析器,DispatchServlet并没有实现任何解析Multipart请求数据的功能，而是委托给MultipartResolver策略接口的实现。Spring内置了两个实现

  • StandardServletMultipartResolver:依赖于Servlet3.0的支持，优先，如果不是Servlet3.0+或者不是Spring3.1+，则选上面的

在Spring应用上下文中，没有构造器参数，也没有要设置的属性，直接声明成bean。

但是上传文件有很多限制，如指定目录，大小等。配置要在Servlet中指定Multipart的配置，具体是在web.xml或者Servlet初始化类中，将multipart作为DispatchServlet配置的一部分。Java配置通过重写customizeRegistration方法来实现，需要extends AbstractAnnotationConfigDispatcherServletInitializer

通过生成MultipartConfigElement来进行配置，其他配置参数有

  • 上传文件的最大容量（字节为单位），默认没有限制
  • 整个multipart请求的最大容量（字节为单位），不会关心有多少个part以及每个part的大小
  • 在上传的过程中，如果文件大小达到指定的最大容量（字节为单位），将会写入到临时文件路径中，默认值为0，也就是所有上传的文件都会写入到磁盘上。

第一步：要Spring 应用上下文中将实体管理器工厂按照bean的形式来进行配置。
基于JPA的应用程序需要使用EntityManagerFactory的实现类来获取EntityManager实例，JPA定义了两个类型的实体管理器

  • 应用程序管理类型，当应用程序箱体管理器工厂直接请求实体管理器时，工厂会创建一个实体管理器并在事务中对其进行控制，适合于不允许在JavaEE容器中的独立应用程序
  • 容器管理类型，实体管理器由JavaEE创建和管理，应用程序根本不与实体管理器工厂打交道，相反，实体管理器直接通过注入或者JDNI来获取。容器负责配置实体管理器工厂，这种类型的实体管理器最适用于JavaEE容器，在这种情况下会希望在persistence.xml指定的JPA配置之外保持一些自己对JPA的控制。

分别由LocalEntityManageFactoryBean生成应用程序管理类型的EntityManageFactory、LocalContainerEntityManagerFactoryBean生成容器管理类型的EntityManagerFactory。

对于应用程序管理类型的实体管理器工厂来说，绝大部分配置信息来源于一个名为persistence.xml的配置文件。这个文件必须位于类路径下的META-INF目录下。

persistence.xml的作用在于定义一个或多个持久化单元。持久化单元是同一个数据源下的一个或多个持久化类。

@Cacheable和@CacheEvict注解。
注解放在单个方法上，缓存行为只会运用到这个方法上，如果注解在类级别，缓存行为就会应用到这个类的所有方法。

  • @Cacheable Spring在调用方法之前，首先应该在缓存中查找方法的返回值，能找到就返回缓存的值，否则的话，这个方法就会被调用，返回值会被放到缓存中
  • @CachePut 表明Spring应该将方法的返回值放到缓存中，在方法的调用前并不会检查缓存，方法始终都会被调用
  • @CacheEvict 表明Spring应该在缓存中清除一个或多个条目
  • @Caching 这是一个分组的注解，能够同时应用多个其他的缓存注解

@Cacheable和@CachePut的共有属性：

  • String[] value 要使用的缓存名称
  • String condition SpEL表达式，如果得到的值是false，不会将缓存应用到方法调用上
  • String key SpEL，用来计算自定义的缓存key
  • String unless SpEL，得到的值是true，返回值不会放到缓存之中

可以将缓存注解放在接口上，而不是实现类上，这样所有实现类都会遵循同样的规则。

@CachePut的目的是，新增一条记录后，很可能很快就用到，所以放在缓存中更快速。

  • #root.args 传递给缓存方法的参数，数组
  • #root.caches 该方法执行时所对应的缓存，数组
  • #root.target 目标对象
  • #root.targetClass 目标对象的类
  • #root.method 缓存方法
  • #root.methodName 缓存方法名
  • #result 方法调用的返回值，不能用在@Cacheable注解上
  • #Argument 任意方法的参数名(#argName) 或参数索引(#a0)

condition和unless属性，如果condition的计算结果为false，不会去缓存查，也不会放入缓存。如果unless计算结果为true，返回的数据不会放到缓存中。

unless只有在有返回值情况下才发挥作用，而condition在进入方法前就需要进行判断

@CacheEvict主要使用在删除数据的情况下，移除缓存

@CacheEvict的属性

  • String[] value
  • String key
  • String condition
  • boolean allEntries 为true，特定缓存的所有条目都会被移除
  • boolean beforeInvocation 为true，在方法调用之前移除条目，为false(默认值),在方法调用之后再移除条目
