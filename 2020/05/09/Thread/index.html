锁的分类

无锁状态、偏向锁、轻量级锁、重量级锁。

锁升级：有竞争。锁降级：monitor池是否空间

对象的对象头里面有个threadid字段，第一次访问时id为空，则线程持有当前对象的锁，成为偏向锁。再次进入时会判断id是否一致，不一致则升级为轻量级锁，通过自旋来获取锁，如果一定次数之后还没获取到，则升级为重量级锁。

定时锁：tryLock(),等待一定时间后根据获取锁立即返回，也能因为别的线程发送中断信号而从阻塞状态恢复。

公平锁：不管有没有线程持有该锁，都会先去阻塞队列，叫公平锁。如果没有线程持有该锁，不会去阻塞队列，而直接获取锁，叫不公平锁。

一个线程因为获取不到锁而阻塞，会被操作系统挂起，即放到硬盘中去，重新恢复就会浪费。

读写锁：一般的锁对变量的读写都是互斥的。但是实际上多个读、一个写是不互斥的，所以有读写锁。

synchronized的三种位置

位置 同步的对象
同步方法块 括号里配置的对象
实例方法 当前实例对象
静态方法 当前类的Class对象

sync修饰静态方法和静态代码块时，锁的是类

sync修饰成员方法时锁的是对象。但是可以同时获得类锁和对象锁

对象锁的实现是JVM利用CAS操作，在指定对象的对象头markword部分设置线程ID，表示当前对象被当前线程占有。

类锁是调用方法区的静态方法，由于是共享的，所以也只能多个线程进行竞争。而对象锁，每个对象都有一个锁，只有针对同一个对象的时候才会产生竞争

每个对象有一个monitor，当monitor被占用时就处于锁定状态。线程执行monitorentey指令时，monitor进入数量+1，如果是重入，则继续+1。其他线程想要获取锁需要等待monitor数量为0才能尝试获取。

每一个monitorexit指令代表monitor数量-1，当为0时线程不再是拥有者，其他线程就可以获取锁。

所以sync的语义是通过monitor对象来完成的

同步方法则是通过常量池是的ACC_SYNCHRONIZED常量来完成的。当调用同步方法时，会检查方法的ACC_SYNCHRONIZED，如果有则获取monitor，获取成功才能执行方法体，其他线程则无法获取同一个monitor对象。

synchronized和valotile的区别

volatile修饰变量，sync修饰类，方法，代码段

volatile只保证可见性和有序性，不保证原子性。sync可以保证可见性和原子性。

volatile不会造成阻塞，sync会造成阻塞

sync和Lock的区别

lock只能给代码块加锁

sync不能手动获取和释放锁，使用简单，异常会自动释放锁。lock需要手动加锁和释放锁

通过tryLock可以知道有没有获取到锁，sync不知道

volatile

通过汇编指令的lock实现

通过缓存一致性和内存指令load read user assgin write store 还有lock unlock
volatile会导致assign–>write–>store为原子
volatile还会禁止重排序
CAS会导致use–>assign为原子的

JVM的线程模型
缓存行的理解：高速缓存大小512字节，主流CPU缓存行一般都是64个字节，因此有8个缓存行。

LRU算法（最近最少使用算法），将最近一段时间最少使用的数据进行淘汰。LRU算法是将每一个缓存行设置一个计数器，命中则清零当前缓存行的值，其他各行计数器加1，需要替换时替换值最大的缓存行。
由于缓存行过小或者正在使用，无法进行替换，就会出现缓存行替换冲突。

可重入锁：如果已经获取锁，再去请求另外一个同步块，也是可以成功的。实现就是进入一个同步块就计数器加一，退出就减一。

成员方法就synchroinzed(this)，静态方法就synchronized(A.class);这样子可以直接在方法上加上synchronized

指令重排序要遵守一定规则，不相互依赖的代码才会被排序

加锁代码块，会让代码块分为前、中、后三个部分，前中后不能被重排序，但是各个部分之内可以被重排序。

在volatile写之前的代码不能被排序到写之后，在读之后的代码不能排序到读之前。前面试volatile写，后面是volatile读不能被重排序。(必须等待写完之后读取更新的值)

Lock和AQS

默认为非公平锁

  • lock的存储结构：一个int类型状态值（用于锁的状态变更），一个双向链表（用于存储等待中的线程）
  • lock获取锁的过程：本质上是通过CAS来获取状态值修改，如果当场没获取到，会将该线程放在线程等待链表中。
  • lock释放锁的过程：修改状态值，调整等待链表。
  • 可以看到在整个实现过程中，lock大量使用CAS+自旋。因此根据CAS特性，lock建议使用在低锁冲突的情况下。目前java1.6以后，官方对synchronized做了大量的锁优化（偏向锁、自旋、轻量级锁）。因此在非必要的情况下，建议使用synchronized做同步操作。

常用方法

  • lock和unlock分别代表获取锁和释放锁，
  • tryLock是获取到立即返回true，没有获取到立即返回false。这样可以根据值来进行调整，比如设置次数，等待时间之后再去获取。这样不会因为获取不到锁而阻塞导致死锁。

进程

进程就是一个程序，最小调度单元。

进程的状态：

  1. 新建：分配进程id，优先级等
  2. 就绪：一旦新建完毕，被放在就绪队列，等待CPU选择执行
  3. 运行：在CPU中运行
  4. 阻塞：可能要进行IO操作，放到阻塞队列中，如果阻塞完成，就放回到就绪队列，等待CPU调用
  5. 退出：执行完毕，或者强行杀死。记录临死前的状态

进程的缺点：

  • 进程之间资源不能共享，信息无法互通
  • 创建、切换、销毁进程的开销很大

最开始进程不仅分配资源，还是调度和执行的基本单位，之后将功能拆分，进程只负责资源的分配单位，线程相当于调度和执行的单位，所以状态就仍然继承线程的状态

进程分配资源，线程获取资源并执行，线程隶属于进程，由于又是同一块资源，所以线程之间可以互相访问。

线程中断

线程中断的流程：线程默认有一个中断标志位，为false。别的线程可以调用t1.interrupt()，将标志位设置为true，而在sleep、join、wait这些方法调用或者阻塞过程中，都会检测标志位是否为true，如果为true则抛出InterruptException，同时将interrupt重新设置为false。

内置锁：如果多个线程竞争，只有一个线程能获取到对象的锁，并且只要不释放锁，其他线程一直等待。

可中断锁的作用：在别的线程里随时让线程在阻塞状态下恢复。

总结

  • 一个程序一个程序执行，切换用户浪费时间（裸机）
  • 解决切换问题，就先将程序写好后，由另外专门负责调度的程序来安排（简单批处理系统）
  • 又由于CPU和内存、内存和外设之间处理速度的差异，在某个程序在进行IO阻塞时，执行另外一个程序，当IO阻塞完成后，再返回去执行之前的程序（多道批处理系统）
  • 将CPU执行时间分为时间片，能达到微秒毫秒级别，所以用户察觉不出来（分时操作系统）

ThreadPoolExecutor

ThreadPoolExecutor执行execute方法有4种情况：
1）如果当前运行的线程少于corePoolSize，则创建新的线程来执行任务。
2）如果运行的线程等于或者多余corePoolSize，则将任务加入BlockingQueue中，在等待队列中，等待有新的线程可以运行。
3）如果BlockingQueue队列满了，且没有超过maxPoolSize，则创建新的线程来处理任务。
4）如果创建的线程超过maxPoolSize，任务会拒绝，并调用RejectExecutionHandler.rejectedExecution()方法。

ThreadLocal用来隔离数据。 ThreadLocal中存放的是与线程相关的数据，底层实际上是一个map，通过线程可以获取存储数据的map。

  • newFixedThreadPool，创建一个固定大小、无界队列的线程池，核心线程=最大线程数
  • newCachedThreadPool，创建一个无界缓冲、存活时间60s，核心线程为0，最大线程为Integer.MAX_VALUE
  • newSingleThreadExecutor()，只有一个线程，无界队列，硬编码创建的线程数
  • newScheduledThreadPool，定时执行任务的线程池，核心线程参数指定，最大线程数为Integer.MAX_VALUE
