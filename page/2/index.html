加载

类加载器

类的加载由类加载器辅助完成。包括BootStrapClassLoader、ExtClassLoader、AppClassLoader。

加载器的顺序

先从自定义ClassLoader查询是否已经加载过，没有则依次往上传递看是否已经加载过。
到达Bootstrap ClassLoader仍然没有加载过，则往下依次在类路径中查找。

加载过程主要完成三件事情：

  • 通过类的全限定名来获取定义此类的二进制字节流
  • 将字节流转换为运行时数据
  • 在堆中生成一个代表此类的java.lang.Class对象

ClassLoader.loadClass源码

加载Class的方式

  • Class.forName()加载类，不但会将类加载，还会执行会执行类的静态变量初始化，因为调用了反射相关的方法。
  • getClass()方法，创建好对象就肯定进行了类初始化。反射可以根据对象获取Class对象，根据对象获取到类信息来进行操作。getClass方法，有多态能力，运行时可以返回子类的类型信息
  • 类名.classs，是静态解析的，编译时可以确定类型信息
  • ClassLoader.loadClass()只加载类，是不会导致类的初始化

自定义类加载

  1. 遵守双亲委派模型，继承ClassLoader，重写findClass()方法
  2. 破坏双亲委派模型，继承ClassLoader，重写loadClass()方法

为什么loadClass()会破坏双亲委派模型？

因为loadClass的源码中优先通过父加载器加载类。如果重写loadClass，则可以修改为先加载自己的类，这样就破坏了双亲委派模型向父类传递的原则。

而findClass在ClassLoader的源码中，loadClass方法在父加载器无法加载到类的情况下，才会调用自身的findClass方法进行加载。所以findClass会先用父加载器加载。

JVM中两个类是否相同,必须是同一个类加载器加载的

虚拟机栈

每个方法被执行都会创建一个栈帧，虚拟机栈就包括多个栈帧。栈帧又分为：

  • 局部变量表，存放各种基本类型和引用类型，占一个变量槽(VariableSlot，32位内存空间)，long和double占2个空间
    • Slot复用。PC计数器的位置超过变量的作用域，对应的Slot就可以交给其他变量使用。能够节省空间，但是影响GC的回收，因为超过了作用域会等待其他变量复用，引用的对象就不会进行回收。
  • 操作数栈
  • 动态连接：每个栈帧持有一个指向运行时常量池所属方法的直接引用。和类加载的静态解析相似，需要知道方法的直接地址，而不是符号引用。
  • 方法出口：方法正常返回可以用调用方法的PC计数器值作为返回地址。对于异常退出则通过异常处理器表来确定。

每个方法被调用完，都代表一个栈帧在虚拟机栈的入栈和出栈

JVM垃圾回收算法

HotSpot虚拟机中存在三种垃圾回收现象，minor GC、major GC和full GC。对新生代进行垃圾回收叫做minor GC，对老年代进行垃圾回收叫做major GC，同时对新生代、老年代和永久代进行垃圾回收叫做full GC。许多major GC是由minor GC触发的，所以很难将这两种垃圾回收区分开。major GC和full GC通常是等价的，收集整个GC堆。

  • V，内存地址
  • A，预期值
  • B，新值

语义为：比较V中的值是否与A相同，相同则将V中的值更新为B，不同则不进行操作（或者返回失败）

String转Byte的问题

如果限制报文长度为1024个字节，也就是1k 10位，那么报文长度的最大值就为1024，所以就需要两个字节16位来表示长度，

  1. 刚开始纠结报文长度可能用两个字节无法表示，比如1024是字符串的话，转换就可能有问题，因为1一个数字就需要两个字节来存储。如果使用“1024”.getBytes()会得到4（一个数字的长度是一个字节，一个字符的长度是两个字节），也就是有四个字节，实际上转换成byte型后，就只需要两个字节就能表示了。

这里仍然有问题，Java中int型是四个字节，所以将int转化为byte数组之后，仍然是四个字节。所以这里实际上还是有问题的。比较可行的是写出的时候int值转byte数组，读取的时候读取byte数组，然后转int。

  1. String转byte会自动在前面加0，读取的时候只需要将byte转换为int，就能得到长度了。
  2. Socket传递的信息能否直接用String，就不需要字节数组转来转去了，应该也是可以的。

UDP单个报文的完整性

如果UDP发送端发送1000个字节，接收端用500个字节接收，就会造成一次性接收到的数据缺失。所以接收端尽量使用大一点的缓冲区接收。

由于UDP是短连接，发送的报文不会太长，所以字节数建议不超过1400。

UDP无法保证完整性和顺序性，Java中可以setData(byte[])来设置缓冲区接收.

UDP还可以组播，使用MulticastSocket。

集合的一些记录

  1. Iterable<E>，实现这个接口允许对象成为 “foreach” 语句的目标。
  2. equals和hashCode都可以用来判断对象是否相等。但是equals一般是给用户调用的，而为了保持逻辑一致，通常在重写equals后，也重写hashCode。
  3. LinkedHashMap和LinkedHashSet做到的只是保证顺序，而不是排序。指的是，查询数据的顺序是按照插入数据的顺序的，但是并不会进行排序。
  4. 集合在迭代遍历时，不可以用集合对象的方法操作集合中的元素，所以一般用iterator。List集合有时需要用listIterator来对对象进行操作。
  5. Collection 都是实现了Iterable的，而Iterable接口的方法iterator会返回一个Iterator迭代器对象，每个对象的游标都是在最开始的位置。
  6. 除了非同步和允许使用 null 之外，HashMap 类与 Hashtable 大致相同。
  7. Object的hashCode主要作用是为了配合基于散列表的集合。如HashMap，HashSet，HashTable。
  8. HashMap中table.length为什么是2的次幂？ JDK7中 HashMap中table.length是2的次幂，为最接近容量2次幂较大值。同时进行hash值映射到table中的时候，使用hash值&length-1来保证key在table范围内。扩容时，由于用的是2*length来扩容，而映射是hash值&length-1,扩大一倍后length-1只是最左边0变为1，那么hash值如果最左边为0，&的操作会让散列到table位置能够和老的保持一致。
  9. HashMap大致了解了put和get的机制，主要是put的机制，扩容、hash运算、检测重复
  10. 别人学习HashMap的步骤：
    • 静态变量和静态代码块
    • 构造函数
    • 常用的方法如put和get
    • 一步步仔细执行，可以先代入值，一步步走
  11. JDK8里面的HashMap源码，final int loadFactor为什么不需要初始化？
  12. 原来早期的计算机取2次幂的余数都是用位运算来计算的。除以8，就是2^3，假设15除以8，那么15（01111）右移三位，就是000 01,这就是15/8的商–1，那么被移除的三位就是00 111（7），即余数。那么既然除以8的时候，我们知道会右移三位。位运算中，X & 1 = X；也就是数和1相与还是原来的数。那么我们只需要将低三位和1与就知道余数了。恰好8-1为1000-0001=0111，也就得到低三位三个1了。就可以直接得到余数了。

HTTP请求的过程

DNS解析

  1. 查看本地缓存是否存在
  2. 本地缓存没有的话查看host文件
  3. host也没有，到最近的DNS服务器查询
  4. 没有会一直到域名根服务器
  5. 全都没有，则报错

HTTP交互四步

  1. 建立TCP连接（三次握手）
  2. 发送请求报文（request）报文组成：请求行、请求头、空行、请求数据
    • 请求行：请求的目标
    • 请求头：客户端的相关信息
    • 空行：请求头的结束
    • 请求数据：
  3. 服务端响应请求（response）
    • 根据GET和POST交给Servlet的doGet和doPost方法来处理
    • 响应报文组成：状态行、响应头、空行、响应数据
      • 状态行：HTTP版本、响应代码、响应描述
      • 响应头：服务器的相关信息、响应数据的信息
      • 空行：响应头的结束
      • 响应数据：HTML文档和图像等，也就是HTML文件
  4. 关闭连接（四次挥手），客户端解析HTML

TCP和UDP

TCP面向连接，UDP发送之前不需要建立连接

TCP可靠，UDP不可靠

TCP通过字节流，UDP通过报文

TCP为什么不两次握手

TCP头部组成中包括：序号（seq），确认号（ack），flags（标记位）

标记位共6个：URG、ACK、PSH、RST、SYN、FIN

  • SYN：建立新连接
  • ACK：确认序号有效
  • FIN：释放一个连接

三次握手的流程：

Client =》 Server：SYN=1，seq=x（客户端建立连接）

Server =》Client：SYN=1,seq=y(服务器建立连接)，ACK=1,ack = seq + 1（确认客户端）

Client =》Server：SYN，seq=x+1，ACK=1，ack=y+1（确认服务端）

服务器发送确认数据包就会建立连接，客户端如果没有相应请求，服务器则会一直等待，白白浪费资源。三次握手则能判断客户端和服务器都已经准备好。

用于定义Web浏览器和服务器之间通讯格式

Http1.0和Http1.1的区别

HTTP1.0协议中，一个连接只能获得一个Web资源，HTTP1.1协议中，一个连接可以获得多个Web资源

  • 一个空行
  • 请求体
